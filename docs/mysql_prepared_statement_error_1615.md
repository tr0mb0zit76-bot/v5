# Ошибка MySQL 1615: "Prepared statement needs to be re-prepared"

## Описание проблемы

Ошибка `SQLSTATE[HY000]: General error: 1615 Prepared statement needs to be re-prepared` возникает в MySQL, когда подготовленное выражение становится недействительным. Это может произойти по нескольким причинам:

1. **Изменение структуры таблицы** во время выполнения подготовленного выражения
2. **Недостаточный размер кэшей** MySQL
3. **Длительные транзакции**, которые удерживают метаданные таблиц
4. **Проблемы с соединением** с базой данных

## Решения, реализованные в приложении

### 1. Исправление в коде приложения

#### a) Метод `syncLegs` в `OrderWizardService.php`
- Заменен прямой `delete()` на `chunkById()` для удаления записей порциями
- Добавлена обработка исключения с переподключением к базе данных при ошибке 1615

#### b) Метод `syncPaymentSchedules` в `OrderCompensationService.php`
- Аналогично заменен прямой `delete()` на `chunk()` с обработкой ошибок
- Добавлен механизм переподключения при ошибке 1615

#### c) Глобальный middleware `ReconnectOnPreparedStatementError`
- Перехватывает ошибки QueryException
- Проверяет, является ли ошибка ошибкой 1615
- Переподключается ко всем соединениям с базой данных
- Повторяет запрос автоматически

### 2. Настройки MySQL (рекомендации для сервера)

Добавьте следующие настройки в конфигурацию MySQL (`my.cnf` или `my.ini`):

```ini
# Увеличить кэш определений таблиц
table_definition_cache = 2000

# Увеличить кэш открытых таблиц  
table_open_cache = 2000

# Увеличить максимальное количество подготовленных выражений
max_prepared_stmt_count = 16382

# Увеличить время ожидания
wait_timeout = 28800
interactive_timeout = 28800

# Оптимизация для InnoDB
innodb_buffer_pool_size = 1G  # или 70-80% от доступной RAM
innodb_log_file_size = 256M
innodb_flush_log_at_trx_commit = 2  # для лучшей производительности
```

### 3. Проверка текущих настроек MySQL

Выполните следующие запросы для проверки текущих настроек:

```sql
-- Текущие значения кэшей
SHOW VARIABLES LIKE 'table_definition_cache';
SHOW VARIABLES LIKE 'table_open_cache';
SHOW VARIABLES LIKE 'max_prepared_stmt_count';

-- Статистика использования
SHOW STATUS LIKE 'Open_table_definitions';
SHOW STATUS LIKE 'Open_tables';
SHOW STATUS LIKE 'Prepared_stmt_count';

-- Рекомендация: если Open_tables близко к table_open_cache, увеличьте значение
```

### 4. Мониторинг и диагностика

#### Команды для диагностики:
```bash
# Показать процессы MySQL
SHOW PROCESSLIST;

# Показать статус InnoDB
SHOW ENGINE INNODB STATUS;

# Мониторинг подготовленных выражений
SELECT * FROM performance_schema.prepared_statements_instances;
```

### 5. Дополнительные рекомендации

1. **Используйте транзакции разумно** - не держите транзакции открытыми слишком долго
2. **Разделяйте DDL и DML операции** - не изменяйте структуру таблиц во время активных операций
3. **Регулярно анализируйте и оптимизируйте таблицы**:
   ```sql
   ANALYZE TABLE order_legs;
   OPTIMIZE TABLE order_legs;
   ```
4. **Используйте индексы** для ускорения операций DELETE:
   ```sql
   -- Убедитесь, что есть индекс на order_id
   SHOW INDEX FROM order_legs;
   ```

### 6. Аварийное восстановление

Если ошибка продолжает возникать:

1. **Перезапустите MySQL**:
   ```bash
   sudo systemctl restart mysql
   ```

2. **Очистите кэши**:
   ```sql
   FLUSH TABLES;
   FLUSH QUERY CACHE;
   RESET QUERY CACHE;
   ```

3. **Временно отключите подготовленные выражения** (если возможно):
   ```php
   // В конфигурации Laravel
   'options' => [
       PDO::ATTR_EMULATE_PREPARES => true,
   ]
   ```

## Заключение

Ошибка 1615 является известной проблемой MySQL, связанной с управлением подготовленными выражениями. Реализованные решения включают:
- Исправление кода для использования chunk-удаления
- Добавление обработки ошибок с переподключением
- Глобальный middleware для автоматического восстановления
- Рекомендации по настройке MySQL сервера

Эти меры должны значительно снизить частоту возникновения ошибки 1615 в production-окружении.
